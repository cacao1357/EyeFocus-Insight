## 1. 项目概览

### 1.1 项目定位

EyeFocus Insight 是一款基于摄像头实时分析面部特征的个人专注度与疲劳监测桌面工具。技术栈为 Python + MediaPipe Face Mesh + OpenCV + SQLite，完全离线运行。

### 1.2 模块完成度

| 模块 | 完成度 | 说明 |
|------|--------|------|
| `detector/face_mesh.py` | ★★★★★ | 完整实现，含线程安全锁、blendshapes 提取、头部姿态矩阵分解 |
| `detector/eye_aspect.py` | ★★★★☆ | EAR 计算+眨眼检测完整，眯眼区分已实现，多信号融合框架在但权重未被动态更新 |
| `detector/head_pose.py` | ★★★★☆ | 矩阵分解正确，稳定性计算完备，但与 face_mesh.py 存在重复的欧拉角提取逻辑 |
| `detector/gaze.py` | ★★★☆☆ | 基础框架完成，但瞳孔位置为近似估算，精度有限 |

## 2. 架构审查

### 2.1 架构优点

1. **模块化清晰** — detector/analyzer/storage/gui 四层分离，职责边界明确
2. **工厂函数模式** — 每个模块提供 `create_xxx()` 工厂函数，统一从 `config.py` 加载默认参数，降低耦合
3. **数据模型统一** — `storage/models.py` 集中定义所有数据类，跨模块共享
4. **回调协议** — `user_calibration.py` 使用 `Protocol` 定义回调接口，解耦校准逻辑与 GUI
5. **三层配置** — config.py(dataclass) → config.yaml → GUI 滑块，优先级清晰

### 2.2 架构问题

#### P1: reporter 模块完全缺失

`reporter/` 目录仅有空壳 `__init__.py`。根据 PROJECT_PLAN.md §2.2，Phase 2 的核心交付物包括：
- HTML 报告生成（`reporter/report_html.py`）
- Matplotlib 图表（`reporter/charts.py`）
- 个性化建议引擎（`reporter/insights.py`）

**影响**：会话结束后无法生成分析报告，用户无法获得跨会话洞察，这是项目的核心价值层缺失。

#### P2: main.py 职责过重（God Class 倾向）

`EyeFocusApp` 类（863行）承担了以下所有职责：
- 模块初始化与编排
- 摄像头线程管理
- 帧处理逻辑（检测→分析→存储流水线）
- 校准流程管理（新旧两套校准系统并存）
- 键盘输入处理
- GUI 渲染参数组装
- 信号处理与退出清理

建议拆分为：
- `CameraManager` — 摄像头采集线程
- `FrameProcessor` — 帧处理流水线
- `CalibrationCoordinator` — 校准流程协调
- `EyeFocusApp` — 顶层编排

#### P3: 两套校准系统并存

代码中同时存在：
1. **旧版** `BaselineCalibrator`（7秒自动采集）— `main.py:199` 中的 `self._calibrator`
2. **新版** `UserCalibrationManager`（6阶段交互校准）— `main.py:241` 中的 `self._calib_manager`

两者在 `main.py` 中交织使用：
- `_process_frame()` 中仍有旧版校准的数据采集逻辑（第466-474行）
- `start()` 方法默认启动新版校准流程
- `_render_frame()` 中同时处理两套校准的 UI 渲染

**风险**：状态不一致、维护负担增加、新开发者困惑。

#### P4: head_pose.py 与 face_mesh.py 欧拉角提取逻辑重复

`face_mesh.py:201-243` 的 `_solve_head_pose_from_matrix()` 和 `head_pose.py:58-113` 的 `detect()` 包含几乎完全相同的矩阵分解代码。

**建议**：在 `face_mesh.py` 中保留提取逻辑，`head_pose.py` 改为接收已提取的 (yaw, pitch, roll) 值进行二次分析。

## 3. 代码质量分析

### 3.1 代码风格

- **一致性**：整体风格统一，遵循 Python 命名规范（类名大驼峰、函数/变量蛇形）
- **文档字符串**：所有公共类和函数都有中文文档字符串，质量较高
- **类型注解**：广泛使用 `Optional`、`Tuple`、`List` 等类型注解，但不完整（如 `main.py` 中部分方法缺少返回值注解）

### 3.2 代码缺陷

#### D1: 工厂函数中的 `or` 陷阱

```python
# face_mesh.py:300-312
return FaceMeshDetector(
    num_faces=num_faces or FACE_MESH.num_faces,
    min_detection_confidence=(
        min_detection_confidence or FACE_MESH.min_detection_confidence
    ),
    ...
)
```

当 `num_faces=0` 或 `min_detection_confidence=0.0` 时，`or` 会错误地回退到默认值。应使用 `if x is None` 检查。

#### D2: eye_aspect.py 中 BlinkEvent 类名冲突

```python
# eye_aspect.py:39 定义
@dataclass
class BlinkEvent: ...

# storage/models.py:68 定义
@dataclass
class BlinkEvent: ...
```

两个不同包中存在同名但不同字段的数据类。`main.py:531` 在写入数据库时进行了手动字段映射，增加了出错风险。

#### D3: baseline.py 中结果计算代码重复

`_finish()` 方法（第223-269行）和 `get_result()` 方法（第271-326行）包含几乎相同的截尾均值+过滤+CQS 计算逻辑。应提取为 `_compute_statistics()` 内部方法。

#### D4: config.yaml 与 config.py 阈值不一致

| 配置项 | config.py | config.yaml | PROJECT_PLAN.md |
|--------|-----------|-------------|-----------------|
| 光照暗阈值 | N/A | 50.0 | 50.0 |
| 光照正常阈值 | N/A | 100.0 | 100.0 |
| 专注度 W_ear | 0.50 | N/A | 0.35 |
| 专注度 W_gaze | 0.25 | N/A | 0.35 |
| 专注度 W_head | 0.25 | N/A | 0.30 |

专注度权重与项目规划文档严重不一致！`focus.py` 使用 (0.50, 0.25, 0.25)，而 PROJECT_PLAN.md 定义为 (0.35, 0.35, 0.30)。

#### D5: CalibrationState 不是真正的枚举

```python
class CalibrationState:
    IDLE = "idle"
    AUTO_CALIB = "auto_calib"
    ...
```

应使用 `enum.Enum` 或 `enum.StrEnum`，当前实现只是类属性字符串常量，无法获得枚举的类型安全优势。

#### D6: user_calibration.py 眨眼检测与 eye_aspect.py 重复

`BlinkRoundCollector.record_frame()`（第125-139行）重新实现了眨眼/眯眼分类逻辑，与 `EyeAspectDetector._update_blink_state()` 功能重叠。两处使用不同的阈值计算方式，可能导致结果不一致。

### 3.3 潜在运行时错误

#### R1: main.py 中 sys.path 修改方式不可靠

```python
# main.py:33
sys.path.insert(0, __file__.rsplit("/", 1)[0] if "/" in __file__ else ".")
```

在 Windows 上 `__file__` 使用 `\` 路径分隔符，`"/" in __file__` 可能为 False，导致回退到 `"."`。虽然功能上可能正常（因为 `"."` 也在 sys.path 中），但逻辑不严谨。

#### R2: FaceMeshDetector.close() 可能无限阻塞

文档注释明确提到 MediaPipe 的 `close()` 可能无限期阻塞（第264行），但仅使用了 `try/except` 而非线程超时机制。`main.py:806-810` 中的 `_cleanup()` 没有对 close 设置超时。

#### R3: 校准 tick 间隔过大

```python
# main.py:383
if current_time - self._last_tick_time >= 1.0:
    self._calib_manager.tick()
```

校准管理器每秒只 tick 一次，但 `_run_auto_calib()` 在每次 tick 中才采集 EAR/姿态数据。30 FPS 的情况下，7秒校准只采集约 7 个数据点，远少于计划中的 150-210 帧。

#### R4: 数据库写入与帧处理在同一线程

虽然 `DatabaseManager` 声称线程安全（使用 `_lock`），但当前架构中帧处理和数据库写入在主循环中串行执行。如果数据库写入变慢（如磁盘 I/O 竞争），会直接拖慢帧处理 FPS。

## 4. 算法与逻辑审计

### 4.1 EAR 眨眼检测算法

**评估：★★★★☆**

- 动态阈值（基线 × 0.75）设计合理，解决固定阈值问题
- 眯眼/眨眼区分（400ms 时间窗口）符合文献
- 多信号融合框架存在但**未被激活**：`_head_pose_weight` 和 `_face_stability_weight` 始终为默认值 1.0，`main.py` 中从未调用 `set_head_pose_weight()` 或 `set_face_stability_weight()`

### 4.2 专注度评分算法

**评估：★★★☆☆**

问题1：**权重与规划文档不一致**（见 D4），实际实现的权重过度偏重眼部（50%），忽视了视线偏离（25%）和头部姿态（25%）。

问题2：**眼部评分函数对眨眼过于敏感**。当用户正常眨眼时（EAR 降至 0.15-0.20），`_compute_eye_score()` 中 `deviation = |0.20 - 0.25| / 0.25 = 0.20`，得分为 `100 - (0.20-0.10)/0.20 * 50 = 75`。每次正常眨眼导致眼部评分从 100 骤降至 75，再通过卡尔曼滤波缓慢恢复。这可能导致专注度评分在正常眨眼时频繁波动。

问题3：**头部评分取 `min(yaw_score, pitch_score)` 过于严苛**。单一维度不佳会完全决定头部评分，应考虑加权平均。

### 4.3 疲劳分级算法

**评估：★★★★☆**

- 基于个人基线的相对倍数判定（1.3x/1.8x）设计合理
- 累积疲劳 EMA（0.95/0.05）时间常数约 20 分钟，符合预期
- 缺失：PROJECT_PLAN.md 中提到的 PERCLOS 指标和"持续 15 秒"条件未在代码中实现

### 4.4 眼镜检测算法

**评估：★★★☆☆**

- blendshapes 眯眼比率方法（阈值 0.85）基于 Phase 0 小样本验证
- **关键缺陷**：`inner_canthus_dist_thresh = 28.0` 是像素距离，与摄像头分辨率强耦合。640×480 分辨率下 28 像素与 1920×1080 下的 28 像素含义完全不同
- 双保险逻辑（两者之一触发即判定）偏激进，可能导致不戴眼镜的用户被误判

### 4.5 光照检测算法

**评估：★★★★☆**

- 五级分类比规划的三级（亮/正常/暗）更精细
- 人脸区域亮度假设人脸位于图像中心 15% 区域，如果用户不在中心位置会不准确
- 平滑和多数投票机制合理

### 4.6 基线校准 CQS 公式

**评估：★★★★☆**

```python
ratio_score = valid_count / total_count * 0.5
cv_score = max(0, (1 - ear_cv * 3)) * 0.5
cqs = min(1.0, ratio_score + cv_score)
```

- `ear_cv * 3` 的乘数 3 意味着 CV 超过 0.333 时 cv_score 为 0，这是一个硬截断
- 校准期间眨眼帧的过滤缺失：`add_frame()` 中未检查当前帧是否为眨眼帧（`blink_flag`），只检查了头部姿态

## 6. 安全与隐私

### 6.1 合规性

- ✅ **完全本地运行**：代码中无任何 HTTP 请求
- ✅ **不存储图像帧**：每帧仅保留数值数据（EAR/yaw/...），blendshapes 以 JSON 存储
- ✅ **SQLite 本地存储**：数据仅在用户主动导出时离开本机

### 6.2 潜在风险

1. **blendshapes JSON 存储**：`frame_records.blendshapes_json` 存储了完整的 52 维 blendshapes 向量。虽然不是图像，但理论上可用于重建部分面部表情信息。对于极端隐私敏感场景，应提供禁用选项。
2. **数据库文件权限**：`data/eyefocus.db` 无额外权限控制，同机用户可读取
3. **摄像头索引硬编码**：默认使用 `index=0`，如果设备有多个摄像头可能意外打开错误的设备

## 7. 性能审查

### 7.1 瓶颈分析

| 瓶颈 | 当前策略 | 评估 |
|------|----------|------|
| MediaPipe 推理 | 单帧处理，无跳帧 | ✅ 合理，CPU 可达 30+ FPS |
| SQLite 写入 | 节流 15 FPS 写帧 + 每秒写疲劳 | ✅ 合理，但批量写入更优 |
| GUI 渲染 | 每帧 cv2.addWeighted 半透明混合 | ⚠️ 不必要的性能开销 |
| blendshapes JSON 序列化 | 每帧 json.dumps | ⚠️ 52 维向量 JSON 序列化/反序列化开销 |
| 摄像头读取 | 独立线程 + Lock | ✅ 合理，避免阻塞主循环 |

### 7.2 性能优化建议

1. **blendshapes 可不存入数据库**：52 维向量写入每帧记录，8 小时会话约 43 万帧 × ~500 bytes = ~200MB 纯 blendshapes 数据。建议仅在需要时计算，或降低存储频率
2. **GUI 半透明混合可简化**：`cv2.addWeighted` 每帧执行多次（状态栏、专注度圆、校准面板），可考虑直接在图像上绘制而不使用 alpha 混合
3. **数据库批量写入**：当前每帧单独 INSERT+COMMIT，改为批量 INSERT（如 15 条一起）可减少事务开销

### 7.3 内存管理

- ✅ 使用 `deque(maxlen=...)` 限制滑动窗口大小
- ⚠️ `EyeAspectDetector._blink_events` 使用无上限 `List`，长时间会话会持续增长。虽然单条记录很小，但 8 小时会话 ~1000 次眨眼，影响有限
- ⚠️ `FocusAnalyzer._recent_data` 的 `maxlen = int(window_size * fps) = int(5.0 * 30.0) = 150`，合理
