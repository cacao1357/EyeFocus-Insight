## 1. 项目核心目标与预期成果

### 1.1 核心目标

本项目的本质不是"又一个眼动追踪 Demo"，而是 **"个人工作状态的可量化自我认知工具"**。

**项目定位**：课程/作品集项目——需要展示完整的工程素养（可运行性、可维护性、可解释性），同时具备一个差异化创新点。

| 层次         | 目标描述                                                                                                 |
| ------------ | -------------------------------------------------------------------------------------------------------- |
| **即时层**   | 实时反馈当前专注度与疲劳程度，辅助用户做出休息/继续工作的决策                                            |
| **会话层**   | 每次会话结束后输出结构化数据，形成可追溯的行为记录                                                       |
| **跨会话层** | 对比多次会话数据，发现个人高效时段、疲劳模式、长期趋势                                                   |
| **洞察层**   | 生成可执行的个性化建议，而不仅仅是"你今天眨眼 3000 次"                                                   |
| **创新层**   | 分心模式识别：自动检测分心行为（频繁视线偏离、头部偏移模式），生成分心热力图帮助用户发现注意力被什么打断 |

### 1.2 关键设计决策（gstack CEO Review 质询后的修正）

基于 Office Hours 质询和 CEO Review 方法论，对初稿做以下方向性修正。初稿中未明确但在规划中需要决策的核心问题，请逐一确认：

| #   | 决策问题                         | 推荐方案                                                 | 理由                                                                                                                              |
| --- | -------------------------------- | -------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| 1   | GUI 框架是 Tkinter 还是 OpenCV？ | **OpenCV highgui 优先，后续可选迁移 Streamlit**          | 满足"实时显示面部关键点 + EAR 曲线"需求，Tkinter 与 OpenCV 视频流集成复杂度高；项目初稿明确 ≥25 FPS 实时性，OpenCV 原生窗口零开销 |
| 2   | 专注度评分算法是启发式还是 ML？  | **加权启发式（可解释优先）**                             | 初稿创新点第 5 条明确要求"每一评分可追溯到具体指标权重"；scikit-learn 标记为"不必须"；启发式规则更符合"可解释性轻量化设计"        |
| 3   | 报告格式 HTML 还是 PDF？         | **HTML 优先（含内嵌 Chart.js/Matplotlib 转 Base64 图）** | PDF 生成需要额外依赖（reportlab/fpdf），HTML 可被任何浏览器打开，且支持交互式图表                                                 |
| 4   | 多用户支持何时实现？             | **Phase 2（MVP 后）**                                    | 初稿提到 user_id，但 MVP 阶段单人使用即可验证核心假设                                                                             |
| 5   | 本地 Ollama 问答何时接入？       | **Phase 3（可选扩展）**                                  | 初稿标记为"可扩展"，不阻塞 MVP 交付                                                                                               |

> ✅ **以上 5 项决策已确认通过，全部采用推荐方案。** 后续规划基于此展开。

## 2. 目标用户群体特征及需求痛点

### 2.1 用户画像

| 画像                   | 特征                                                       | 使用场景                                                 |
| ---------------------- | ---------------------------------------------------------- | -------------------------------------------------------- |
| **知识工作者（核心）** | 程序员、设计师、写作者，每天面对屏幕 6-12 小时             | 想知道自己"真正专注的时间有多少"，以及何时应该休息       |
| **远程办公者**         | 在家/咖啡厅工作，缺乏办公室社交监督，自我管理依赖工具      | 量化自己的工作节奏，对抗"假勤奋"（坐在电脑前但效率低下） |
| **学生/研究者**        | 长时间阅读文献、写论文，容易出现视疲劳但不自知             | 防止长时间用眼导致的视力下降，建立健康的学习习惯         |
| **量化自我爱好者**     | 已有运动手环、睡眠监测等习惯，希望补上"认知状态"的数据拼图 | 将专注度数据与其他健康数据交叉分析                       |

### 2.2 核心需求痛点

```
┌─────────────────────────────────────────────────────┐
│  痛点 1："我不知道我什么时候真正在专注"               │
│  表现：一天过去感觉疲惫但说不清干了什么                │
│  根因：缺乏客观的专注度度量工具                        │
│  本项目的解法：EAR + 头部姿态 + 视线偏离 → 专注度评分   │
├─────────────────────────────────────────────────────┤
│  痛点 2："我感觉眼睛很累，但不知道多累算'太累'"        │
│  表现：等到眼睛干涩/头痛时才意识到，已经晚了            │
│  根因：没有实时疲劳监测和预警                          │
│  本项目的解法：眨眼率变化 + 闭眼时长 → 疲劳分级预警      │
├─────────────────────────────────────────────────────┤
│  痛点 3："我的高效时段是早上还是下午？我不确定"         │
│  表现：凭感觉安排工作时间，经常选错时间段               │
│  根因：没有跨会话的趋势分析                            │
│  本项目的解法：跨会话历史追踪 → 高效时段标注             │
├─────────────────────────────────────────────────────┤
│  痛点 4："市面上的工具要么太贵，要么把我的摄像头数据      │
│          传到云端，我不信任"                           │
│  表现：拒绝使用需要联网的眼动追踪工具                   │
│  根因：隐私顾虑 + 付费墙                               │
│  本项目的解法：完全本地运行，SQLite 数据不出本机         │
└─────────────────────────────────────────────────────┘
```

### 2.3 用户旅程映射

```
[启动程序] → [5-8秒基线校准] → [实时监测(主循环)]
    → [每帧: 人脸检测 → EAR计算 → 头部姿态 → 专注度评分 → GUI更新 → 数据写入SQLite]
    → [用户按Q退出] → [自动生成HTML报告] → [查看报告]
    → [下次启动: 加载历史数据 → 对比分析]
```

#### 专注度评分算法设计（可解释性核心）

```
专注度评分 = 100 - clamp( W1 × f1(EAR偏离) + W2 × f2(视线偏离) + W3 × f3(头部不稳), 0, 100 )

其中：
  W1 = 0.35  (眨眼异常权重)      — 疲劳主要通过眨眼模式体现
  W2 = 0.35  (视线偏离权重)      — 不看屏幕 = 不专注
  W3 = 0.30  (头部不稳权重)      — 频繁晃动 = 注意力分散
  注：初始权重基于设计假设，需经 Action Item #4 真人测试调优

滑动窗口设计（避免等级滞后与抖动）：
  f1(EAR偏离)  = |当前EAR(EMA α=0.1) - 基线EAR| / 基线EAR × 100
  f2(视线偏离) = 过去10秒视线偏离帧数 / 总有效帧数 × 100   （每0.5秒刷新）
  f3(头部不稳) = 过去10秒头部角度(yaw/pitch/roll)滚动标准差 / 15° × 100

EMA 平滑：专注度最终值 = 0.15 × 瞬时评分 + 0.85 × 上一帧输出  (τ≈6.7秒)

GUI 底部信息栏示例：
  专注度: 68  [EAR偏离: -3  |  视线偏离: -18  |  头部不稳: -11]
  三项扣分之和 = 32，对应 100 - 32 = 68 分，用户一眼看懂扣分来源
```

#### 基线校准算法（三级过滤 + 截尾均值）

```
校准数据流（采集 5-8 秒，约 150-240 帧）：

Step 1: 去除眨眼帧 — blink_flag=1 的帧直接丢弃
Step 2: 去除大幅偏转帧 — |yaw| > 10° 或 |pitch| > 10° 的帧丢弃
Step 3: 对剩余帧取 10% 截尾均值（trimmed mean）

校准质量评分 (CQS)：
  IF 帧丢弃率 > 40%:  CQS = FAIL → 提示"检测到异常动作，请重新校准"
  IF EAR标准差 > 0.05: CQS = WARN → 提示但允许继续
  IF 头部角度标准差 > 5°: CQS = WARN → 提示但允许继续
  ELSE: CQS = PASS → 自动进入监测

动态漂移补偿（仅用于疲劳判定，不用于专注度评分）：
  baseline_ear(t) = 0.95 × baseline_ear(t-1) + 0.05 × recent_avg_ear
  限制：若 recent_avg_ear < baseline_ear × 0.7（真疲劳），跳过本次更新
```

#### 头部姿态 3D 模型（solvePnP 6 锚点）

```
3D 参考点（通用人脸模型归一化坐标）：
  鼻尖     (0.0,  0.0,  0.0)    ← MediaPipe 索引 1
  下巴     (0.0, -7.3, -1.5)    ← MediaPipe 索引 152
  左眼角   (-3.3,  1.8,  2.0)   ← MediaPipe 索引 33
  右眼角   (3.3,   1.8,  2.0)   ← MediaPipe 索引 263
  左嘴角   (-2.4, -4.2,  1.5)   ← MediaPipe 索引 61
  右嘴角   (2.4,  -4.2,  1.5)   ← MediaPipe 索引 291
  注：solvePnP 天然处理尺度差异（通过 tvec 范数），无需手动归一化
```

#### 疲劳分级量化阈值

```
正常:   眨眼率 < 基线 × 1.3  AND  PERCLOS(闭眼时长占比) < 5%
轻度:   眨眼率 ≥ 基线 × 1.3   OR  PERCLOS ≥ 5%  (持续 15 秒以上)
重度:   眨眼率 ≥ 基线 × 1.8   OR  PERCLOS ≥ 10% (持续 15 秒以上)

PERCLOS = 最近 60 秒内闭眼(blink_flag=1)总时长 / 60
  注："持续 15 秒"条件防止喷嚏/揉眼导致的瞬时误报
```

#### 眼镜/遮挡自动降级

```
触发条件（校准时检测）：
  IF 校准阶段 EAR 方差 < 0.003（经验阈值，说明关键点僵直/被遮挡）:
    → 标记 glasses_mode = True
    → W1(眨眼权重) 降为 0.10, W2/W3 按比例上调至 0.45/0.45
    → GUI 角标显示 [👓] 图标
    → SQLite 记录字段 glasses_mode = 1
```

#### 数据库 Schema

```sql
CREATE TABLE IF NOT EXISTS sessions (
    session_id   TEXT PRIMARY KEY,         -- UUID
    user_id      TEXT DEFAULT 'default',
    start_time   TIMESTAMP,
    end_time     TIMESTAMP,
    pause_start  TIMESTAMP,               -- 长时间检测中断的开始时间
    pause_end    TIMESTAMP,               -- 中断结束时间
    glasses_mode INTEGER DEFAULT 0,       -- 0=正常, 1=眼镜降级模式
    baseline_ear REAL,
    baseline_ear_std REAL,                -- 校准阶段 EAR 标准差（用于眼镜检测）
    baseline_blink_rate REAL,
    baseline_yaw REAL,
    baseline_pitch REAL,
    baseline_roll REAL
);

CREATE TABLE IF NOT EXISTS frames (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL,
    timestamp    REAL NOT NULL,            -- 相对于会话开始的秒数
    left_ear     REAL,                     -- NULL 表示检测丢失
    right_ear    REAL,
    blink_flag   INTEGER DEFAULT 0,        -- 0/1
    perclos      REAL,                     -- 最近 60 秒闭眼时长占比
    yaw          REAL,
    pitch        REAL,
    roll         REAL,
    gaze_status  TEXT,                     -- 'screen' / 'away' / NULL(丢失)
    fatigue_label TEXT,                    -- 'normal' / 'mild' / 'severe' / NULL
    focus_score  REAL,                     -- 0.0 ~ 100.0, NULL 表示检测丢失
    focus_breakdown TEXT,                  -- JSON {"ear_penalty":3, "gaze_penalty":18, "head_penalty":11}
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);

CREATE INDEX idx_frames_session ON frames(session_id, timestamp);
```

### 4.3 性能设计

| 瓶颈            | 策略                                                                                                 |
| --------------- | ---------------------------------------------------------------------------------------------------- |
| MediaPipe 推理  | 每帧推理，不跳帧；目标 25+ FPS（实际 MediaPipe Face Mesh 在笔记本 CPU 上可达 30+ FPS）               |
| SQLite 写入     | 每 0.5 秒批量写入一次，而非逐帧写入；使用 WAL 模式提升并发；内存缓冲区（deque）暂存，SIGINT 时先刷盘 |
| Matplotlib 出图 | 仅在会话结束时生成图表，不在主循环中调用                                                             |
| 内存            | 每帧关键点数据仅在内存中保留最新值，历史数据落入 SQLite                                              |

### 4.4 并发与线程安全（关键设计修正）

> ⚠️ **重要修正**：原计划提到"GUI 在主线程，计算可在子线程"，但经过技术审查，MediaPipe 官方不推荐多线程使用。因此**全线采用单线程模型**。

```
单线程主循环（每帧）：
  1. cap.read()                    → 获取帧
  2. face_mesh.process(frame)      → MediaPipe 推理
  3. eye_aspect(head_pose, ...)    → EAR/姿态计算（纯数值运算，微秒级）
  4. analyzer.update(...)          → 评分/分级（纯数值运算）
  5. buffer.append(record)         → 缓冲区追加
  6. IF len(buffer) >= 15:         → 每 0.5 秒批量写入 SQLite
  7. gui.render(frame, overlays)   → OpenCV imshow
  8. atexit.register(flush_buffer) → 异常退出时刷盘

缓冲区设计：collections.deque(maxlen=1000)，内存安全
```

### 4.5 缺失数据处理策略

```
人脸检测失败时的帧处理：
  ┌─ 单帧丢失（< 2 秒连续）：
  │   写入 SQLite，但 left_ear/yaw 等为 NULL（保留 timestamp 和 session_id）
  │   专注度评分沿用最近有效值（hold-last-value 策略）
  │
  ├─ 连续丢失 2-10 秒：
  │   写入 NULL 记录，GUI 显示 "未检测到人脸"
  │   专注度评分线性衰减至 0（约 5 秒衰减周期）
  │   报告生成时将此段标注为 "无效段（检测中断）"
  │
  └─ 连续丢失 > 10 秒：
       自动暂停会话，记录 pause_start / pause_end 到 sessions 表
       恢复检测后继续监测，基线不变（但报告标注中断段）
```

### 4.6 分心模式识别算法（差异化创新核心）

```
分心检测的数据输入（来自主循环连续帧）：
  每帧记录: gaze_status (screen/away), yaw_degrees, pitch_degrees, timestamp

分心事件定义：
  ┌─ "短暂分心"：gaze_status='away' 持续 3-15 秒
  │   → 记录为一次 distraction_event (type='short')
  │
  ├─ "中长分心"：gaze_status='away' 持续 15-60 秒
  │   → 记录为一次 distraction_event (type='medium')
  │
  └─ "长分心"：gaze_status='away' 持续 > 60 秒
      → 记录为一次 distraction_event (type='long')，可能触发暂停

分心热力图生成规则：
  1. 按分钟聚合：每一分钟计算 (away_seconds / 60) = 分心率
  2. 颜色映射：0%(绿) → 50%(黄) → 100%(红)
  3. 在 HTML 报告中渲染为横向热力条，横轴 = 会话时间轴
  4. 跨会话分析：统计每天各时段（9:00-9:30, 9:30-10:00...）的平均分心率
     → 发现"上午 10 点容易分心看手机"等模式

分心模式分类：
  ┌─ "高频短促型"：短分心事件 > 10 次/小时 → 可能在看消息/刷社交媒体
  ├─ "间歇中长型"：中长分心事件 3-5 次/小时 → 可能在切换任务
  └─ "单次长断型"：出现 > 5 分钟的长分心 → 可能在休息或用手机深度操作

报告中的呈现：
  分心热力图（时间轴颜色条）
  + 分心事件列表（时间、类型、估计原因）
  + 个性化建议："你的高频分心时段为 10:00-10:30，建议该时段安排不需要深度专注的任务"
```

| 子问题                                     | 回答                                                                                                                                                                                |
| ------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Q1: 权重是否基于文献？是否对人种差异敏感？ | 初始权重(W1=0.35, W2=0.35, W3=0.30)基于设计假设，非文献。但自适应基线校准解决人种 EAR 差异问题（亚洲人 EAR≈0.25，高加索人≈0.35），因为评分基于**偏离个人基线的幅度**而非绝对值。    |

| Q3: 用户能否理解"为什么是68分"？           | GUI 底部栏实时显示 `专注度: 68 [EAR偏离:-3 \| 视线偏离:-18 \| 头部不稳:-11]`，三项扣分之和=32。HTML 报告中含权重说明章节。SQLite `focus_breakdown` 字段存储 JSON 扣分明细，可追溯。 |

| Q3: 基线是否应动态微调？            | **缓慢漂移补偿**：`baseline_ear = 0.95×old + 0.05×recent_avg_ear`（时间常数≈20分钟），仅补偿光照漂移。若 recent_avg_ear < baseline×0.7（真疲劳），跳过更新。动态基线仅用于疲劳判定，专注度评分始终用初始基线以保持跨会话可比性。 |

<summary><b>A5 — 疲劳分级阈值</b>（🟡中风险）</summary>

| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Q1: 偏离基线多少触发"轻度"/"重度"？  | 正常: 眨眼率<基线×1.3 AND PERCLOS<5%。轻度: 眨眼率≥基线×1.3 OR PERCLOS≥5%（持续15秒以上）。重度: 眨眼率≥基线×1.8 OR PERCLOS≥10%（持续15秒以上）。"持续15秒"防止瞬时误报。 |
| Q2: 是否结合 PERCLOS？               | **是**。PERCLOS（Percentage of Eyelid Closure）= 最近60秒闭眼总时长/60。它是疲劳监测的金标准指标，与眨眼率互补。                                                          |
| Q3: 固定阈值对眨眼频率低者是否误报？ | 阈值是基于**偏离个人基线的倍数**，不是绝对值。天生眨眼率 10次/分钟的人，阈值是 13次/分钟（×1.3），而非和 20次/分钟的人共用同一阈值。                                      |

</details>

| 子问题                                | 回答                                                                                                                        |
| ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Q1: 虹膜边界点在眨眼/侧脸时是否稳定？ | Phase 3 才实现。MediaPipe 468 点中虹膜边界点约 8-12 个，在侧脸、眯眼时确实不稳定。EPA 仅作为辅助指标，不参与核心评分。      |
| Q2: EPA 与 EAR 的相关性？             | 两者高度相关（都反映眼睑开合程度），EPA 的优势在于对部分遮挡更鲁棒。若相关性>0.85，EPA 提供的信息增量有限，可能不值得投入。 |

</details>
